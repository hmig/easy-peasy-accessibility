The Audit

  • accessibility evaluation of the home pages for the top 1,000,000 web sites (from the Majestic Millions list)
  • used WAVE accessibility engine to analyze the rendered home pages

The folks of WebAIM ran the WAVE accessibility engine on the top 1 million home pages.

The Million

The million comes from "the majestic million" - This is basically any/every major site you can think of...and then almost a million more.

Wave

The WAVE engine uses logic to detect patterns in web page content and matches those patterns against the Web Content Accessibility Guidelines (WCAG) conformance to detect failures.

Automation can only catch around 25% of WCAG conformance failures.

Here's the thing though ... All automated tools are limited in their ability to detect accessibility issues. and can only detect approximately 25% of possible conformance issues. So, getting 0 errors or 100% does not mean that a site is accessible or compliant.

That being said, the data from this audit can provide a meaningful representation of the state of web accessibility.

Accessibility audit reveals:

  • 97.8% of home pages had detectable WCAG 2 failures
  • an average of 59.6 errors per page
  • error density - 1 in every 13 html elements

Audit the Site

The only way to find and fix accessibility issues is to audit the site.

Prioritize

  1. Site-wide elements (header, footer, nav, sidebar)
  2. High priority reusable page/layout templates
  3. High priority reusable components (toggles, buttons, modals, ...)
  4. High priority pages (homepage, product pages, cart, ...)
  5. Repeat 2-4 with the lower priority items

First thing to do is to prioritize where to start. It makes sense to focus on site-wide elements like the header, footer and nav AND high priority reusable page templates and components.

Probably the easiest way to do this is to focus on high priority pages like the homepage and a few other specific pages that check off a bunch of these list items just by fixing a few pages, it can have a large overall impact. Repeat.

Think "POUR"

The four main guiding principles of accessibility are:

Perceivable - provide content alternatives (images, audio, video)

Operable (without a mouse/trackpad)

Understandable - clear and simple (writing and functionality)

Robust - works across many devices

Keyboard Testing

Once you have the results of the automated test for a page, then you'll want to test it for keyboard navigation.

Tabbing.

tabbing

  • Do you have to go through the main nav? Add a skip nav (and make sure it works!) -- add tabindex=-1 to elements you want to be focusable, but not in the normal tab order
  • does the page highlight what your tabbing to?
  • Do NOT remove the default :focus { outline: none; outline: 0; }! Update the css if you do not like how it looks, but do not remove it! It is imperative for ppl that nav via keyboard to see what the focus is on.
  • Can you access all of the actionable elements?

Weird Tabbing?

The DOM order does not match the visual order

DO NOT add a tabIndex that is greater than 0.

What does the graphic add to the page?

Does it look like stock photography, space filler or meaningless eye candy?

If it is purely decorative, we do not need to add alternative text.

We still need the alt attribute on the image BUT we can leave it empty [no space]. If you put a space, some screen readers will stumble on those images instead of skipping over them.

For images that have text around them, does the surrounding content serve as the alternative text?

In this case - yes.

Images are sometimes added to a text link to increase the clickable area or to make it look better. Again, no alt text needed.

Functional Images

Functional images are used to initiate actions rather than to convey information. They are used in buttons, links, and other interactive elements.

The text alternative for the image should convey the action that will be initiated (the purpose of the image), rather than a description of the image.

Other common accessibility issues

So, that concludes the list from the audit, but there are other common issues that pop up.

Semantic HTML

  • Headings
    • are meant to help users understand content hierarchy
    • levels should not be skipped (BAD - h1, h2, h4)
  • Links, Buttons, Divs and Spans
    • a, div and span ARE NOT buttons 
      Products
    • if it acts like a button, make it a

Headings

  1. Heading tags are for content hierarchy (outline), not for styling
  2. If you want an h3 to look like an h4, style it with CSS, but don't skip heading levels.

I see a lot of Divs and spans being used as links and buttons...

  1. DIVs and SPANs are not buttons
  2. and are not natively supported to have keyboard focus, name, role, or value ... why use them and have to add all of this support via more code?
  3. If you don’t like how a button or link looks, style it with CSS.
  4. If the element navigates, it is a link. Use link markup with a valid hypertext reference
  5. If the element triggers an action, it is a button. Use a BUTTON element

Accessible Rich Internet Applications (ARIA)

ARIA: provides a set of attributes that you can add to HTML elements.

  • role
  • state
  • property

ARIA was created by the Web Accessibility Initiative (or WAI) of the W3C and it provides a standard that allows developers to set:

  • roles
  • states
  • and properties

on html elements.

First rule of ARIA use: Don't use it!

If you can use a native HTML element or attribute, do it. - straight from the w3c

"ARIA roles add nothing to default semantics of most elements" - Steve Faulkner

Don't add default implicit roles to elements

❮button role="button"❯ button text ❮/button❯


Don't add ARIA state or property attributes in addition to their native HTML counterparts

❮input type="text" required aria-required="true"❯

❮div hidden aria-hidden="true"❯


Don't add ARIA roles and states or properties to long-implemented structural elements

❮h1 role="heading" aria-level="1"❯heading text❮/h1❯

HTML5 Elements with implicit semantics

What this means is that, where implemented, the browser will expose the default implicit semantics of the element so you don’t have to.

❮header❯ maps to role="banner"

❮nav❯ maps to role="navigation"

❮main❯ maps to role="main"

❮section❯ maps to role="region"

❮article❯ maps to role="article"

❮aside❯ maps to role="complementary"

❮footer❯ maps to role="contentinfo"
